# NGAIP-365 - Implement Context Relevancy Metric

## Ticket Purpose

NGAIP-365 measures whether retrieved context is relevant to each gold question/reference answer. It owns the retrieval-context quality metric implementation that plugs into the `NGAIP-363` harness and reports according to the `NGAIP-415` contract.

## Runbook Decision

Use RAGAS context metrics as the primary signal. Keep token overlap as a secondary diagnostic because the original acceptance criteria referenced chunk overlap and because deterministic checks are useful for CI smoke tests.

Primary metrics:

- RAGAS context precision
- RAGAS context recall

Secondary diagnostics:

- token overlap at K
- hit rate at K
- retrieval precision/recall at K when deterministic gold spans or chunk ids are available

## LanceDB Context Source

The contexts scored by RAGAS should come from the existing LanceDB-backed retrieval path. Use the same embedding model/deployment that populated LanceDB; mismatched embeddings will make context relevancy scores misleading.

## RAGAS Role

RAGAS is used because semantically relevant context may not share exact tokens with the gold answer. The metric should ask whether the retrieved context can support the expected answer, not only whether words overlap.

The implementation should adapt PrattWise harness outputs into RAGAS input shape:

- question or user input
- contexts retrieved from the existing LanceDB vector store
- reference answer or gold answer
- optional retrieved metadata for reporting

## Deterministic Role

Keep deterministic overlap metrics for:

- quick CI checks without model credentials
- debugging retrieval regressions
- compatibility with the original chunk-overlap acceptance criteria
- cases where `gold_chunk_ids` or exact spans are available from `NGAIP-362`

Token overlap should not be the final semantic quality gate unless stakeholders explicitly choose it after calibration.

## Expected Files

This ticket should update the context metric module generated by the harness, typically under:

- `app_retrieval/evaluation/metrics/context_relevancy.py`
- `tests/app_retrieval/test_context_relevancy.py`

It should consume shared helpers from:

## Source Modality Handling

For PDFs, OCR, tables, and mixed source parts, this metric should score the text/markdown context that PrattWise retrieved and passed to the assistant. It should preserve `content_type` and source metadata in the result so table/OCR failures can be analyzed separately.

If KG context is attached by `NGAIP-362`, it may be included in the context evaluated by RAGAS, but the report should make that provenance visible.


CI-safe tests should mock RAGAS/model calls and verify:

- deterministic token overlap still works.

- report output includes evaluator metadata when live RAGAS is enabled.

Live RAGAS tests should be opt-in and require Azure OpenAI credentials.

## RAGAS-Primary Update

`NGAIP-365` should use RAGAS context precision/recall/relevancy metrics as the primary context relevancy signal. Token overlap remains useful for fast CI diagnostics and threshold calibration, but it is no longer the main metric.

## Reasoning, Choices, and Code Breakdown

The main design choice is to make RAGAS context precision/recall the primary context relevancy signal while retaining token overlap as a fast diagnostic. Semantic context can be relevant even when it does not share many words with the reference answer, so overlap alone is not a reliable quality gate.

Rejected alternatives:

- Token overlap as the primary metric: cheap but too brittle for semantic retrieval.
- Manual review only: useful for calibration, but too slow for repeatable CI/regression checks.
- Running against a separate toy retriever: would not measure the existing LanceDB-backed PrattWise retrieval behavior.

Code/file breakdown:

- `metrics/context_relevancy.py`: thin metric module that builds/uses RAGAS context metrics through the shared adapter and exposes deterministic overlap diagnostics.
- `tests/test_context_relevancy.py`: verifies RAGAS input shaping, empty retrieval handling, deterministic overlap behavior, and evaluator metadata plumbing.
- `harness.py` from `NGAIP-363`: supplies retrieved contexts from semantic, keyword, or hybrid retrieval paths.
- `sample_gold.jsonl` from `NGAIP-362`: gives a small fixture for validating context conversion before real corpus data is ready.

This choice aligns the metric with the user-visible RAG experience: the question is whether retrieved context can support the answer, not whether the same exact words appear.
